Houston×Hermes Agent

Should Hermes Agent replace the Claude Code / Codex / Gemini CLIs as the “brain” behind Houston? A grounded look at what fits, what fights, what we keep, and the proxy in the middle.

Prepared for the Houston engine team Scope: engine/ provider layer + wire protocol Verdict: additive, not wholesale

⊕ Bottom line

Don’t make Hermes the wholesale desktop “brain.” The CLIs Houston shells out to are doing two jobs at once — they are the LLM client and the agent harness (tool loop, file edits, session state). Hermes is also both. So “use Hermes instead of the CLIs” is not swapping an LLM client; it is swapping one entire agent operating system for another that overlaps ~70% with what the Rust engine already owns (sessions, skills, memory, cron, tools, its own SQLite store).

Recommended shape: adopt Hermes additively as one more ProviderAdapter (an HTTP adapter instead of a subprocess one) HTTP, and make it the strategic home for the server-side products (Always On / Teams) where a heavyweight autonomous Python agent is an asset rather than a liability. Keep the CLIs for the desktop default. This reuses Houston’s existing “adding a provider” recipe almost verbatim and bets nothing.

The proxy you intuited is real and necessary — but it already exists. It’s the houston-engine wire protocol (/v1/* REST + /v1/ws firehose). The frontend (desktop, mobile, examples) never talks to a provider directly; it only talks to that contract. Whatever the brain is, that contract stays fixed. See §6.

1How Houston’s brain works today

Houston is a Tauri desktop app whose React frontend talks only to a standalone Rust binary, houston-engine, over HTTP+WS. The engine is what actually drives a model: for every conversation turn it spawns a provider CLI subprocess, streams its NDJSON output back, and persists the result.

What the engine builds per provider

Each provider is a ProviderAdapter in a static REGISTRY (provider/mod.rs). A turn is routed through session_dispatch.rs to a provider-specific runner that assembles a command line:

  • Claude — claude -p --output-format stream-json --include-partial-messages --system-prompt … --model … --resume …; prompt on stdin.
  • Codex — codex exec --json -c developer_instructions={…} -c model_reasoning_effort=… --model … resume …; prompt on stdin.
  • Gemini — gemini --output-format stream-json --yolo --resume …; system prompt wrapped in <system>…</system> on stdin.

What the engine owns around the call

  • Streaming: three bespoke NDJSON parsers (parser.rs, codex_parser.rs, gemini_parser_state.rs) reassemble deltas into typed FeedItems and push them onto the WS firehose.
  • History: libSQL chat_feed table (durable, unbounded), keyed by session id; replayed via /history.
  • Agent state: files-first — .houston/<type>/<type>.json (activity, routines, config, learnings) + a notify file-watcher that turns agent-side writes into events.
  • Provider + model: per-agent in .houston/config/config.json; switched live via PATCH /v1/workspaces/:id/provider.
  • Scheduling: houston-scheduler fires cron routines → starts a normal provider turn.
  • Errors: a typed ProviderError taxonomy with per-provider stderr/result classifiers; every failure becomes a user-visible card (no silent failures).
  • Distribution: the signed .app bundles codex+composio+gemini (~940 MB) and runtime-downloads proprietary claude-code; Windows also extracts a portable git-bash.

The load-bearing insight: the CLI is not just “the model.” It is Houston’s tool-execution sandbox (Edit/Write/Bash run in the agent’s working directory, which is exactly why the file-watcher reactivity works), its auth surface (subscription OAuth via claude auth login / codex login), and its streaming protocol. Replacing it touches all of those at once.

2What Hermes Agent actually is

Hermes Agent (Nous Research, Python 3.11 + Node 22, MIT, github.com/NousResearch/hermes-agent) is a complete autonomous agent runtime — not a model gateway. It happens to expose an OpenAI-compatible HTTP server, but behind that server is a full agent OS.

The API server OpenAI-compatible

Enabled via ~/.hermes/.env (API_SERVER_ENABLED=true, API_SERVER_KEY=…), started with hermes gateway, listening on 127.0.0.1:8642. Bearer-token auth. Endpoints:

  • POST /v1/chat/completions — standard Chat Completions, SSE streaming with custom hermes.tool.progress events.
  • POST /v1/responses — server-side conversation state via previous_response_id / named conversations. Caps at 100 stored responses (LRU).
  • POST /v1/runs → run_id; GET /v1/runs/{id}/events (SSE tool-progress + tokens); POST /v1/runs/{id}/stop.
  • GET /v1/models, GET /v1/capabilities, GET /health.
  • /api/jobs — CRUD for scheduled / background runs (pause / resume / manual trigger).

Key constraint: over the API, “the model field is cosmetic; the actual LLM is configured server-side.” You change models by changing the profile, not per request.

The agent OS behind it

  • Providers via API keys in ~/.hermes/.env: openrouter, nous, openai, anthropic, copilot, deepseek, gemini, minimax, xai, ollama-cloud, bedrock, azure-foundry, custom (OPENAI_BASE_URL). Three API modes: chat_completions, codex_responses, anthropic_messages.
  • Tools: 70+ tools across ~28 toolsets (terminal, browser, web, file, vision, MCP), executed on one of 7 terminal backends (local, Docker, SSH, Daytona, Modal, Singularity, Vercel Sandbox).
  • Skills: ~/.hermes/skills/, agentskills.io-compatible SKILL.md — the same open standard Houston uses. Skills Hub pulls from skills.sh, GitHub, /.well-known/skills, custom taps.
  • Memory: MEMORY.md + USER.md in ~/.hermes/memories/, plus a self-improvement loop and state.db SQLite + FTS5 session_search.
  • Multi-agent: profiles — hermes -p <name> gives each agent its own HERMES_HOME, config, memory, sessions, and gateway PID. Concurrent. Shareable as “profile distributions.”

The uncomfortable symmetry

Read that second column again next to §1. Hermes ships its own version of nearly everything the Houston engine already implemented in Rust: sessions, durable store, skills (same standard), memory/learnings, cron jobs, MCP, multi-agent isolation, tool execution. Hermes is not a missing piece that slots into Houston — it is a parallel Houston with a different runtime and a different opinion about who owns state.

4Pros — the genuine wins

① Kills the CLI-bundling problem

Today the .app carries ~940 MB of CLIs, a runtime claude-code download, a per-arch Composio fork build, a Windows git-bash extraction, and “Gemini is macOS-only in v1.” Talking to provider APIs via keys deletes that entire apparatus and its release-CI signing/notarization burden.

② Many more providers/models

From 3 CLIs to a dozen API providers (OpenRouter, Nous, DeepSeek, xAI, Bedrock, Azure, Ollama, custom OpenAI-compatible endpoints). Massive reach for power users with one credential model.

③ One streaming protocol, not three

The three bespoke NDJSON parsers collapse into a single SSE consumer. codex_parser.rs, gemini_parser_state.rs, and the Claude stream accumulator are a real maintenance tax that one Hermes adapter would retire.

④ Shared skills standard

Both sides speak agentskills.io SKILL.md. Houston’s skills are largely portable to Hermes and vice-versa — the integration doesn’t fork the ecosystem.

⑤ Philosophical alignment

Hermes is MIT and open-source, like the Houston engine. It can be forked, pinned, or embedded. And its autonomy (memory, self-improvement, jobs) is a strong match for the Always On 24/7 vision.

⑥ Still reaches Claude & Codex

The anthropic_messages and codex_responses API modes mean adopting Hermes does not drop access to the models users already want — it just reaches them over the API instead of the CLI.

5Cons & risks — the frictions

① A Python+Node runtime on the desktop

The engine is one pure-Rust sidecar binary. Hermes needs Python 3.11, Node 22, uv, ripgrep, ffmpeg. Bundling that on a non-technical user’s laptop is arguably heavier than today’s CLIs — you’d trade three binaries for a full interpreter environment. This mostly makes sense only server-side.

② Two agent OSes, one product

Hermes owns sessions, skills, memory, cron, tools, and a store. So does Houston. You must either (a) demote the engine to a thin proxy and let Hermes own state — abandoning Houston’s files-first reactivity and .houston/ identity — or (b) keep Houston owning state and run Hermes with its own features disabled, fighting it the whole way.

③ “Model is cosmetic” vs. live model switching

Houston’s UX is per-agent provider+model, switchable live. Over Hermes’ API, model is server-side config. To preserve the picker you’d run one Hermes profile/port per (provider, model) — a process fleet — or reconfigure-and-restart. Neither matches today’s per-turn --model flag.

④ Loss of subscription sign-in

Today users authenticate with their Claude / ChatGPT subscriptions via CLI OAuth — no API key, no metered bill. Hermes is API-key-first. For Houston’s explicitly non-technical audience, “paste an API key and pay per token” is a real onboarding + cost regression. (Verify whether Hermes supports subscription OAuth for Anthropic/OpenAI — §9.)

⑤ Reactivity depends on local file writes

Houston’s “the agent edited your project, UI updates instantly” loop relies on tool writes landing on the local filesystem in the agent dir, caught by the watcher. Hermes’ Docker/SSH/Daytona/Modal backends execute elsewhere. You must pin Hermes to the local backend rooted exactly at the agent working dir, or the reactivity story silently breaks.

⑥ Maturity & single-vendor coupling

Today’s CLIs come from Anthropic/OpenAI/Google with relatively stable contracts. Hermes is young, fast-moving, and from one lab. Betting the product’s “brain” on it couples Houston’s reliability and release cadence to Hermes’.

⑦ The clean win doesn’t actually require Hermes

The most attractive pro — “kill CLI bundling, get many providers via API keys” — is achievable with a direct Rust HTTP provider adapter (Anthropic Messages / OpenAI / OpenRouter) plus Houston’s own tool loop, without importing a second agent runtime. Hermes’ distinctive value is its autonomy, tools, and memory — and those only pay for themselves server-side, where you actually want a long-lived self-improving agent. Be honest about which problem you’re solving before paying Hermes’ integration cost.

6What we keep no matter what

This is the reassuring part of your question. The frontend is insulated from the brain by design. Regardless of which option we pick, these stay exactly as they are:

  • The entire houston-engine wire protocol — /v1/* REST (17 route modules: workspaces, agents, sessions, files, routines, skills, store, preferences, providers, …) and the /v1/ws firehose of HoustonEvent. This is the proxy you described. The desktop app, mobile PWA, and the smartbooks example consumer all speak only this. It is the stable interface; the brain is an implementation detail behind it.
  • Files-first reactivity — .houston/ data surfaces + the notify file-watcher + TanStack-Query invalidation. This is Houston’s identity and must keep owning state.
  • The ProviderError → FeedItem → toast surfacing path — the no-silent-failures policy. Hermes errors get classified into the existing taxonomy; the frontend cards are unchanged.
  • The product system prompt + non-technical voice (houston_prompt.rs) — still injected, now as Hermes instructions rather than a CLI --system-prompt flag.
  • Workspaces / agents / Store / agent-manifest — the organizing model above the brain.

In other words: the work is entirely behind the POST /v1/agents/:path/sessions boundary. Above it, nothing changes. Below it, the runner stops spawning a CLI and starts driving Hermes.

7Three integration options

Risky

A · Wholesale brain swap

The engine keeps its wire contract but every turn is delegated to Hermes for all providers. The CLIs are removed.

Cost: ship/host Python+Node; resolve the double-OS state ownership; rebuild auth around API keys; rebuild model-switching as a profile fleet.

Effort: very high · Risk: high · Reversibility: low
Recommended

B · One more provider adapter

Add provider/hermes.rs as an HTTP adapter beside the subprocess ones. Users point an agent at a Hermes endpoint (local or remote). Claude/Codex/Gemini CLIs stay untouched.

Cost: one adapter file + registry entry + 3 dispatch arms (the existing recipe) + an SSE→FeedItem parser + error classifiers.

Effort: moderate · Risk: low · Reversibility: high
Strategic

C · Server-side engine option

Run Hermes on the VPS for Always On / Teams, where a 24/7 autonomous Python agent is a feature and a Python runtime is normal. A thin shim fronts it with the Houston wire protocol.

Cost: the §8 translation layer, deployed server-side; no desktop bundling.

9Open questions to verify before committing

  1. Subscription OAuth? Can Hermes authenticate Anthropic/OpenAI via user subscriptions (Claude Pro / ChatGPT Plus), or is it API-key-only? This decides whether the desktop cost/onboarding story regresses for non-technical users. Likely the single biggest go/no-go for Option A.
  2. Reasoning/thinking deltas. Does the SSE stream expose reasoning tokens distinctly, so we can keep the thinking UI? If not, that surface degrades.
  3. Per-request model. Is “model is cosmetic” truly absolute, or is there a header/param to override per run without a profile? This decides whether live model switching needs a process fleet.
  4. Pinning the local terminal backend to the exact agent working dir, so file writes land where the watcher sees them (reactivity).
  5. Tool-call fidelity. Does hermes.tool.progress carry enough structure (tool name, args, result) to render Houston’s tool cards, and to compute the post-turn file_changes feed item?
  6. Concurrency model. Houston runs many sessions across many agents in parallel; how many concurrent runs does one Hermes gateway sustain, and does that force the per-agent profile/process fleet?
  7. Resource footprint & startup time of a bundled Python+Node Hermes vs. today’s sidecar — for the desktop case only.

10Recommendation & sequencing

Do this

  1. Spike Option B now. Build provider/hermes.rs as an HTTP adapter against a locally-run Hermes (hermes gateway), wire the §8 mappings, and let a single agent target it. Low risk, fully reversible, proves the streaming + error translation against a real Hermes. This satisfies “use Hermes” without betting anything.
  2. Answer the §9 questions during the spike — especially subscription OAuth and per-request model. They determine whether Option A is even viable for the desktop audience.
  3. Make Hermes the brain for Option C (Always On / Teams), not the desktop. Server-side is where Python is fine and where a long-lived, self-improving, tool-rich agent is a genuine product advantage. The same §8 layer deploys as a shim there.
  4. Treat Option A as a future decision, gated on §9. If subscription OAuth is unavailable, A regresses the core non-technical-user value prop — and the “many providers, no CLI bundle” win is better captured by a direct Rust API adapter than by adopting a second agent OS.

One-line summary: Houston already has the right seam (the provider registry + the wire protocol). Use it. Add Hermes as a provider and as a server-side engine — don’t dissolve Houston’s engine into a Hermes proxy.
